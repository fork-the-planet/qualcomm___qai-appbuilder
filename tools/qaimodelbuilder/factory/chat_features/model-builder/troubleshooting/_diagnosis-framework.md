# 通用诊断方法论总纲（Universal Diagnosis Framework）

> **知识定位**：任何领域排障子 SKILL（转换/推理/精度/性能/稳定性……）的**统一开篇总纲**——规定"怎么诊断"，各领域 SKILL 规定"这个领域的症状与修法"。
> **抽取内核**：四阶段（根因调查 → 模式分析 → 单一假设最小验证 → 改根因不改症状）、三条铁律、反向追溯（沿调用链回溯到原始触发点）、多层加固、红旗清单与合理化对照表。
> **剥离外壳**：某调试方法论框架的内部 skill 名引用、语言专属示例改写为语言/领域无关的伪代码骨架。
> **说明**：本文件名以 `_` 前缀标记为"共享总纲/非独立可触发 SKILL"，故**无 YAML front-matter**，不会被 skill 发现器当作独立 SKILL 加载；各 troubleshooting 子 SKILL 在开篇引用它。

## 核心原则

胡乱试错浪费时间还制造新 bug；打补丁只是盖住底层问题。

**核心原则：动手修之前，永远先找到根因。修症状 = 失败。违反本流程的字面，就是违反诊断的精神。**

## 铁律（The Iron Law）

```
未完成根因调查，不许提任何修复方案
```

## 何时用

任何技术问题：测试失败、生产 bug、行为异常、性能问题、构建失败、集成问题、API 返回错误码……**尤其在**有时间压力、"就一个小修改"看起来很显然、已试过好几个修复、上一个修复没生效、你并不完全理解这个问题时。别在"问题看起来简单/在赶时间/上级要马上修好"这些借口下跳过。

---

## 三条铁律（贯穿全程，务必操作化）

1. **未完成根因调查，不许提修复**（= 阶段 1 未完成前，禁止给方案）。
2. **一次只改一个变量**（最小验证；不要一把改多处，否则无法定位哪个生效、还会引入新 bug）。
3. **同一问题修了 3 次还没好 = 这是架构问题，必须停下报告**，不许盲目发起第 4 次修复。

---

## 四阶段（必须逐阶段完成，不得跳阶段）

### 阶段 1：根因调查（动手修之前）

1. **仔细读错误信息**：别跳过 error/warning，它们常直接含答案；读完整栈；记下行号/路径/错误码。
2. **稳定复现**：能否可靠触发？确切步骤？每次都发生吗？不可复现 → 收集更多数据，不要猜。
3. **查近期改动**：什么改动可能导致？git diff / 近期提交 / 新依赖 / 配置变化 / 环境差异。
4. **多组件系统里埋点收集证据**：提修复前，先在每个组件边界加诊断埋点——记录进入/离开该组件的数据、校验环境/配置是否正确透传、检查每层状态；先跑一次看清"在哪一层断的"，再只深入那个组件。
5. **反向追溯数据流**（错误深在调用栈里时）：沿调用链回溯到坏值的**源头**，在源头修，不在症状点修（见下文 §反向追溯）。

### 阶段 2：模式分析（修之前先找规律）

1. **找可用的正例**：同代码库里类似的、能跑的代码。
2. **对照参考实现**：若在套用某模式，**完整读**参考实现，别只扫一眼。
3. **列出差异**：正常 vs 异常之间**每一处**不同，再小也列。
4. **理解依赖**：它还需要哪些组件/配置/环境？做了什么假设？

### 阶段 3：假设与测试（科学方法）

1. **形成单一假设**：明确写下"我认为根因是 X，因为 Y"，具体、不含糊。
2. **最小验证**：做**能验证该假设的最小改动**，一次一个变量。
3. **验证后再继续**：生效 → 阶段 4；没生效 → 形成**新**假设，**别**在旧修复上叠加更多修复。
4. **不懂就说不懂**：去查、去问，别假装懂。

### 阶段 4：实施（改根因，不改症状）

1. **先建失败用例**：最简复现（自动化测试最好），**修之前必须有**。
2. **只做单一修复**：针对已定位的根因，一次一个改动，不夹带"顺手改进"、不捆绑重构。
3. **验证修复**：现在测试过了吗？没弄坏别的测试吗？问题真解决了吗？
4. **修复没生效时**：**停**，数一下试了几次——<3 次回阶段 1 带新信息重新分析；**≥3 次停下质疑架构**。
5. **3 次以上失败 = 质疑架构**：征兆是每个修复在不同地方揭示新的共享状态/耦合、都要大重构才能落、在别处制造新症状。停下质疑根本模式是否站得住，**在发起更多修复前先与人讨论**。

---

## 反向追溯（Root-Cause Tracing）

bug 常显现在调用栈深处，本能是在报错处修，但那是治标。沿调用链**向后**追到原始触发点，在源头修：① 观察症状；② 找直接原因；③ 问"谁调用了它？"一层层往上列调用链；④ 继续往上看传进来的是什么值；⑤ 找到那个坏值最初从哪产生，在那里修。手动追不动时，在**危险操作发生之前**（不是失败之后）打日志，带上下文（关键参数、当前工作目录/状态、环境变量、时间戳）并抓完整调用栈。**原则：绝不只在错误显现处修。**

---

## 多层加固（Defense-in-Depth）——找到根因后让 bug 结构上不可能复现

只在一处加校验，会被不同代码路径 / 重构 / mock 绕过。**在数据流经的每一层都加校验**。四层模板：① 入口点校验（API 边界拒绝明显非法输入）；② 业务逻辑校验（数据对本操作有意义）；③ 环境守卫（特定上下文禁止危险操作）；④ 调试埋点（危险操作前记录上下文+调用栈）。与"修复要连根拔"呼应：找到一处根因后系统排查同类问题、每层加固。

---

## 红旗清单 — 出现即停、回阶段 1

若你发现自己在想："先快修一下之后再查" / "随便改改 X 看看行不行" / "一次加多个改动跑一下测试" / "跳过测试我手动验一下" / "八成是 X 我先修那个" / "我不完全懂但这个也许能行" / "模式说要 X 但我换个方式套" / 还没追数据流就在提方案 / **"再试最后一个修复"（其实已试 2 次以上）** / 每个修复都在不同地方揭示新问题——**全部意味着：停，回阶段 1**。已 3 次以上失败则质疑架构（阶段 4.5）。

## 合理化对照表

| 借口 | 现实 |
|---|---|
| "问题简单，不用走流程" | 简单问题也有根因，流程对简单 bug 很快。 |
| "紧急，没时间走流程" | 系统化诊断比猜-试乱撞更快。 |
| "先试这个，之后再查" | 第一个修复定了基调，一开始就做对。 |
| "确认修复有效后再补测试" | 未测的修复不牢，先写测试才能证明。 |
| "一次修多个省时间" | 无法隔离哪个生效，会引入新 bug。 |
| "参考太长，我套一下模式就行" | 半懂必出 bug，完整读。 |
| "我看到问题了，直接修" | 看到症状 ≠ 理解根因。 |
| "再试最后一个修复"（已 2 次以上失败） | 3 次以上失败 = 架构问题，质疑模式，别再修。 |

## Quick Reference

| 阶段 | 关键动作 | 通过标准 |
|---|---|---|
| **1. 根因** | 读错误、复现、查改动、埋点收证、反向追溯 | 弄清 WHAT 和 WHY |
| **2. 模式** | 找正例、对照、列差异 | 识别出差异 |
| **3. 假设** | 形成单一假设、最小验证 | 确认或换新假设 |
| **4. 实施** | 建失败用例、单一修复、验证、多层加固 | bug 解决、测试通过、结构上防复现 |

> **"查不到根因"时**：若系统化调查确实表明问题是环境性/时序性/外部因素，① 已走完流程 ② 记录查过什么 ③ 实施恰当处理（重试/超时/明确错误信息）④ 加监控/日志。**但：95% 的"没有根因"其实是调查不彻底。**
