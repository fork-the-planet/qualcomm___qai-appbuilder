# Environment Troubleshooting (base)

> 🧭 通用诊断骨架（四阶段 + 三铁律 + 反向追溯 + 多层加固）见 [`../_diagnosis-framework.md`](../_diagnosis-framework.md)；本 SKILL 是该总纲在"环境/依赖/构建"领域的症状库。

## Responsibility

Fix WoS ARM64 environment problems: VS ARM64 / MSBuild (Community vs BuildTools) failures, the
two-Python-environment split (x64 3.10 conversion vs ARM64 3.13 inference), missing inference deps
(`cv2`, `Pillow`, `qai_appbuilder`), `protobuf` conflicts, and PyTorch `--index-url` rules. All paths
come from `${APP_ROOT}\data\config\qairt_env.json` (generated by `Setup.bat`) — **never hardcode**.

## Trigger signals

- `No CMAKE_C_COMPILER` / `CMAKE_C_COMPILER not found`
- `VCTargetsPath.vcxproj` / `BaseOutputPath not set` / `MSB1009` / `Failed to run MSBuild`
- `ModuleNotFoundError: No module named 'qai_appbuilder' / 'cv2' / PIL`
- `ImportError: cannot import name 'builder' from 'google.protobuf.internal'`
- Unclear which Python env a tool needs → **B10**

## Core knowledge

### Two Python environments — NOT interchangeable

| Env | Key in `qairt_env.json` | Python | Role |
|-----|-------------------------|--------|------|
| Conversion | `python_x64_venv` | x86_64 3.10 | ONNX export, `qnn-onnx-converter`, `qnn-model-lib-generator` |
| Inference | `python_arm64_venv` | ARM64 3.13 | `qai_appbuilder`, `QNNContext` inference |

**Default rule for tools NOT in SKILL.md:** default to `python_x64_venv` (most QAIRT-adjacent tools link `python310.dll` → fail under ARM64 3.13 with `Module use of python310.dll conflicts with this version of Python`). Only use `python_arm64_venv` when the tool imports `qai_appbuilder`/`QNNContext` or is pure inference on a prebuilt `.bin`/`.dlc`. Still uncertain → **B10** (stop, ask). Real example: `GenAIBuilderFactory`/`libPyNetRun.pyd` links `python310.dll` → must use `python_x64_venv`.

### VS ARM64 / MSBuild issues

`qnn-model-lib-generator` and `qnn-context-binary-generator.exe` require the VS ARM64 build env.

- **`CMAKE_C_COMPILER not found`** — `vcvarsall.bat arm64` was not called, or `VCTargetsPath` → BuildTools. Fix: use `run_pipeline.bat` (auto), or in a custom `.bat` read `vs_vcvarsall` + `vc_targets_path` from `qairt_env.json` and `call "%_VCVARSALL%" arm64` in the same process (env does NOT propagate across `cmd /c "..."`).
- **`VCTargetsPath.vcxproj` / `BaseOutputPath not set` / `MSB1009`** (Platform='ARM64') — `VCTargetsPath` points to VS 2022 **BuildTools**; BuildTools MSBuild cannot compile ARM64 DLLs. Fix: set `VCTargetsPath` to **Community** (`vc_targets_path` in `qairt_env.json` already does). Verify: `where.exe MSBuild.exe` must show `...\2022\Community\MSBuild\...`, NOT `(x86)\...\BuildTools\...`.

Correct paths:
- ✅ `C:\Program Files\Microsoft Visual Studio\2022\Community\MSBuild\Microsoft\VC\v170\`
- ❌ `C:\Program Files (x86)\Microsoft Visual Studio\2022\BuildTools\...`

### `qai_appbuilder` import error

`ModuleNotFoundError: No module named 'qai_appbuilder'` → use ARM64 Python from `python_arm64_venv`:
```bat
%PYTHON_ARM64% -c "import qai_appbuilder; print('OK')"
```
If it still fails → run `Setup.bat` from the QAIModelBuilder project directory. Offline install (if not on PyPI): `data\bin\uv\uv.exe pip install vendor\whl\qai_appbuilder-*.whl`.

### Missing inference deps (`cv2`, `Pillow`)

On a properly-installed ARM64 venv these are pre-installed by `Setup.bat` from `vendor\whl\`:

| Package | Status | Notes |
|---------|--------|-------|
| `numpy` | pre-installed | do not reinstall |
| `opencv-python-headless` (`import cv2`) | pre-installed | headless; `cv2.imshow` unavailable. **Do NOT install `opencv-python` (GUI build) — conflicts** |
| `Pillow` (`from PIL import Image`) | via editable `pip install -e .` | official win_arm64 cp313 wheels |
| `torch`/`torchvision` | **not** pre-installed | see `--index-url` rule |

If `import cv2`/`PIL` fails, `Setup.bat` didn't complete — re-run it, do NOT `pip install opencv-python` (introduces a conflicting second OpenCV).

### `protobuf` conflict

`ImportError: cannot import name 'builder' from 'google.protobuf.internal'` → `onnx` and `tensorflow` have conflicting `protobuf` requirements; install scripts pin compatible versions. **Fix: re-run `Setup.bat`.** Do not manually install packages outside the install scripts.

### PyTorch / torchvision `--index-url` rules

| Env | Package | Command |
|-----|---------|---------|
| `python_x64_venv` | torch/torchvision | `...python.exe -m pip install torch torchvision` — **no** `--index-url` (win_amd64 wheels on PyPI) |
| `python_arm64_venv` | torch/torchvision | `uv.exe pip install torch torchvision --index-url https://download.pytorch.org/whl` — **MUST** (no ARM64 Win wheels on PyPI) |
| `python_arm64_venv` | all other packages | `uv.exe pip install <pkg>` — **no** `--index-url` |

Using `--index-url` when not needed breaks other installs. If weight download fails with `SSLCertVerificationError`, use `Invoke-WebRequest` to fetch the `.pth`, then re-run export (torchvision uses the cache).

### Architecture detection

Do NOT use `$env:PROCESSOR_ARCHITECTURE` / `platform.machine()` (emulation-affected). Use `(Get-WmiObject Win32_Processor).Architecture` (12=ARM64, 9=x64) or `dumpbin /headers model.dll | find "machine"`. Never use `Get-PnpDeviceProperty` / `Get-WmiObject Win32_SystemDriver` for SoC detection — they wake the DSP / hit WMI throttling and hang 300-400s.

### Verify env quickly
```powershell
$cfg = Get-Content ${APP_ROOT}\data\config\qairt_env.json | ConvertFrom-Json
& "$($cfg.python_x64_venv)\Scripts\python.exe"  -c "import onnx; print('conversion env OK')"
& "$($cfg.python_arm64_venv)\Scripts\python.exe" -c "import qai_appbuilder; print('inference env OK')"
```

## Related Blocking Conditions

- **B10** — a needed tool/package is not described in SKILL.md and the correct venv is unclear → **STOP, ask the user.** Do not default to ARM64 just because the goal is inference; the right env depends on what the tool links against internally.
- **B2** — a `pip install` is needed → stop, state package + reason, ask permission.
- **B1** — a required config variable in `qairt_env.json` is empty/placeholder → stop, list missing vars.

## Escalation path

If `Setup.bat` re-run does not restore a missing env/dep, or the required venv for a tool cannot be determined → stop and report (B10/B1/B2). Never silently guess a Python env or install unpinned package versions.

Full setup details, tool-path rules, and SoC/HTP-version detection → `references/win_qairt_setup.md` and SKILL.md § "Python Environment Management".
