# QNN Conversion Reference

## Scope
Use this reference when `{FLOW}=QNN`.

> ЁЯЪж **Path selector тАФ read this first**
>
> Two QNN conversion pipelines exist. Choose based on the user's request:
>
> | Situation | Use | Script |
> |-----------|-----|--------|
> | **Default / all new work** | **Flow A** тАФ ONNX тЖТ DLC тЖТ .bin | `run_pipeline.py` |
> | User explicitly asks for the DLL pipeline / needs the `.dll` artifact | **Flow C (legacy)** тАФ ONNX тЖТ C++/BIN тЖТ DLL тЖТ .bin | `run_pipeline_legacy.py` |
>
> The **default `run_pipeline.py`** in this codebase now runs Flow A. It invokes `qairt-converter` тЖТ optional `qairt-quantizer` тЖТ `qai_dev_gen_contextbin.py --model <file>.dlc`. No VS ARM64 env required.
>
> The sections below (Verified Flow, per-step commands, `qai_convert_fp.py` / `qai_convert_int.py`) document the **legacy Flow C** implementation for reference / debugging. When invoked via `run_pipeline_legacy.py` they still work end-to-end. Newly written docs for the DLC pipeline live in the SKILL.md Core Workflow Step 4 and in `--help` output of `run_pipeline.py`.

> **Wrapper scripts vs. direct toolchain calls**
> `qai_convert_fp.py` and `qai_convert_int.py` are the **recommended entry points for the legacy Flow C** (DLL-based path). They internally invoke `qnn-onnx-converter` and `qnn-model-lib-generator`, and include **tested auto-detection logic** that resolves known QAIRT toolchain path issues across all supported host architectures (ARM WIN / X86 LINUX / ARM LINUX).
> `--preserve_io` is passed automatically to keep input/output tensor order consistent with the ONNX model.
> Prefer these scripts over calling the toolchain binaries directly.
> For the default Flow A pipeline, use `run_pipeline.py` instead тАФ it wraps `qairt-converter`/`qairt-quantizer` with the same care.

## тЬЕ Complete Verified Flow (Legacy Flow C тАФ QAIRT 2.45, WoS ARM64, FP16)

> This section documents the **legacy DLL-based Flow C**. For the default DLC pipeline (Flow A), read `run_pipeline.py --help` and the Core Workflow Step 4 in the parent `SKILL.md`.

The following 3-step flow has been verified on Snapdragon X Elite (V73) with QAIRT 2.47.1.260610.

> ЁЯТб **`.cpp` patching**: In this verified scenario, no `.cpp` patching was required. However, patching
> **may still be needed** for other models or configurations тАФ always follow the operator patching guide
> (`operator_patching.md`) if context binary generation fails with `Wrong number of Parameters` or
> `Op validation failed` errors.
>
> ЁЯТб **Model preparation**: Before exporting to ONNX, disable training-only branches (e.g., `aux_logits`).
> These branches may contain operators that QAIRT 2.45 cannot convert.
> See `references/model_export_validation.md` for the general pattern.

### Prerequisites

> тЪая╕П **Python environment**: Read paths from `${APP_ROOT}\data\config\qairt_env.json` (auto-generated by `Setup.bat`).
> - Conversion (Steps 1тАУ2): use `python_x64_venv` (x86_64 Python 3.10)
> - Inference (Step 4+): use `python_arm64_venv` (ARM64 Python 3.13)
> - Do **not** hardcode absolute paths тАФ they vary per machine.

```bat
@echo off
REM Read all paths from ${APP_ROOT}\data\config\qairt_env.json тАФ do NOT hardcode
for /f "delims=" %%S in ('powershell -NoProfile -Command "(Get-Content ${APP_ROOT}\data\config\qairt_env.json | ConvertFrom-Json).qairt_sdk_root"') do set QNN_SDK_ROOT=%%S
for /f "delims=" %%P in ('powershell -NoProfile -Command "(Get-Content ${APP_ROOT}\data\config\qairt_env.json | ConvertFrom-Json).python_x64_venv"') do set PYTHON_X64=%%P\Scripts\python.exe
for /f "delims=" %%V in ('powershell -NoProfile -Command "(Get-Content ${APP_ROOT}\data\config\qairt_env.json | ConvertFrom-Json).vs_vcvarsall"') do set VS_VCVARSALL=%%V
for /f "delims=" %%T in ('powershell -NoProfile -Command "(Get-Content ${APP_ROOT}\data\config\qairt_env.json | ConvertFrom-Json).vc_targets_path"') do set VCTargetsPath=%%T

set PYTHONPATH=%QNN_SDK_ROOT%\lib\python;%PYTHONPATH%
set PATH=%QNN_SDK_ROOT%\lib\aarch64-windows-msvc;%QNN_SDK_ROOT%\bin\aarch64-windows-msvc;%QNN_SDK_ROOT%\bin\x86_64-windows-msvc;%PATH%

REM Initialize VS 2022 Community ARM64 (REQUIRED for Step 2)
call "%VS_VCVARSALL%" arm64
```

### Step 1: ONNX тЖТ C++/bin

```bat
REM тЪая╕П CRITICAL: The stem of --output_path determines the graph name used in Step 3!
REM   --output_path output/<model>.cpp  тЖТ  graph name = "<model>"
REM   Use the model name as stem for clarity.

%PYTHON_X64% "%QNN_SDK_ROOT%\bin\x86_64-windows-msvc\qnn-onnx-converter" ^
    --input_network ${WORKSPACE}\<model>\<model>.onnx ^
    --no_simplification ^
    --output_path ${WORKSPACE}\<model>\output\<model>.cpp ^
    --float_bitwidth 16 ^
    --preserve_io
```

**Expected output:**
```
INFO - Model CPP saved at: output/<model>.cpp
INFO - Model BIN saved at: output/<model>.bin
INFO - Conversion complete!
```

> ЁЯТб `--float_bw 16` is deprecated but still works (equivalent to `--float_bitwidth 16`).
> ЁЯТб `--no_simplification` suppresses ONNX simplification warnings and is recommended for WoS.
> ЁЯТб ONNX opset 18 (from PyTorch 2.x) is fully supported тАФ no downgrade needed.
> ЁЯТб Large models may produce `.onnx` + `.onnx.data` (external data format) тАФ pass the `.onnx` path, converter finds `.data` automatically.

### Step 2: C++/bin тЖТ ARM64 DLL

```bat
REM тЪая╕П Uses aarch64-windows-msvc (NOT x86_64) тАФ this is the most common mistake
REM Requires: vcvarsall.bat arm64 + VCTargetsPath = VS 2022 Community (set in Prerequisites above)
%PYTHON_X64% "%QNN_SDK_ROOT%\bin\aarch64-windows-msvc\qnn-model-lib-generator" ^
    -c ${WORKSPACE}\<model>\output\<model>.cpp ^
    -b ${WORKSPACE}\<model>\output\<model>.bin ^
    -o ${WORKSPACE}\<model>\output ^
    -t windows-aarch64
```

**Expected output:**
```
INFO - Library name         : <model>
INFO - Generated model library path: output\ARM64\<model>.dll
```

### Step 3: DLL тЖТ Context Binary

```bat
REM Copy HTP runtime files AND run generator with CWD = output dir
REM (qai_dev_gen_contextbin.py handles both steps automatically)
copy "%QNN_SDK_ROOT%\lib\aarch64-windows-msvc\QnnHtp.dll"            ${WORKSPACE}\<model>\output\
copy "%QNN_SDK_ROOT%\lib\hexagon-v73\unsigned\libqnnhtpv73.cat"       ${WORKSPACE}\<model>\output\
copy "%QNN_SDK_ROOT%\lib\hexagon-v73\unsigned\libQnnHtpV73Skel.so"    ${WORKSPACE}\<model>\output\

REM CRITICAL: qnn-context-binary-generator.exe resolves libqnnhtpv73.cat and
REM libQnnHtpV73Skel.so relative to its CWD -- must cd to output dir first.
REM qai_dev_gen_contextbin.py passes cwd=output_dir automatically.
REM Use --auto-config to generate backend_extensions.json automatically.
%PYTHON_X64% ${APP_ROOT}\factory\chat_features\model-builder\scripts\qai_dev_gen_contextbin.py ^
    --model ${WORKSPACE}\<model>\output\ARM64\<model>.dll ^
    --output_dir ${WORKSPACE}\<model>\output ^
    --binary_file <model>_fp16 ^
    --auto-config
```

**Expected output (success):**
```
Starting stage: Graph Preparation Initializing
...
Starting stage: Completion
Completed stage: Completion (515 us)
```
тЖТ Output file: `${WORKSPACE}\<model>\output\<model>_fp16.bin`

> ЁЯТб `--auto-config` generates `backend_extensions.json` and `htp_backend_config_v73.json` automatically.
> `graph_names` in the config is set to the DLL's graph name (= stem of `--output_path` from Step 1).
> Pass `--config_file` instead of `--auto-config` to use a custom config.

### Configuration Files (manual creation, if not using --auto-config)

> тЪая╕П **`graph_names` MUST match the stem of `--output_path` in Step 1.**
> e.g., `--output_path output/<model>.cpp` тЖТ `graph_names` must be `["<model>"]`.
> Wrong `graph_names` тЖТ `unable to find graphName` error.

**`backend_extensions.json`** (place in working directory):
```json
{
  "backend_extensions": {
    "shared_library_path": "<qairt_sdk_root>\\lib\\aarch64-windows-msvc\\QnnHtpNetRunExtensions.dll",
    "config_file_path": "${WORKSPACE}\\<model>\\output\\htp_backend_config_v73.json"
  }
}
```

**`htp_backend_config_v73.json`** (Snapdragon X Elite V73):
```json
{
  "graphs": [
    {
      "graph_names": ["<model>"],
      "vtcm_mb": 8,
      "O": 3
    }
  ],
  "devices": [
    {
      "cores": [
        {
          "rpc_control_latency": 100,
          "perf_profile": "burst"
        }
      ]
    }
  ]
}
```

> Replace `<qairt_sdk_root>` with the value from `${APP_ROOT}\data\config\qairt_env.json`.
> Use double backslashes `\\` in JSON string paths on Windows.

### Common Errors Quick Reference

| Error | Cause | Fix |
|-------|-------|-----|
| `unable to find graphName` | `graph_names` in config тЙа DLL graph name | Use `--auto-config`; or set `graph_names` = stem of `--output_path` |
| `MODEL_INVALID_ARGUMENT_ERROR` | Same as above | Same fix |
| `Graph Compose failure` | Config mismatch or unsupported op | Check graph_names; check operator support |
| `No CMAKE_C_COMPILER` | VS ARM64 env not initialized | Call `vcvarsall.bat arm64` in same `.bat` file |
| `Unable to load backend: QnnHtp.dll` | DLL not in PATH or working dir | Copy `QnnHtp.dll` to working dir |
| `Backend version mismatch` | Using wrong SDK version's QnnHtp.dll | Use same SDK version for all steps |
| `Wrong number of Parameters 5` | Missing VS ARM64 environment | Ensure running inside a `.bat` file with `vcvarsall.bat arm64` called at top. Also ensure `libqnnhtpv73.cat` and `libQnnHtpV73Skel.so` are in working dir. If still fails after correct env, patch `.cpp` per `operator_patching.md` |
| `VCTargetsPath.vcxproj` / `BaseOutputPath not set` | `VCTargetsPath` points to **BuildTools** instead of **Community** | Read `vc_targets_path` from `${APP_ROOT}\data\config\qairt_env.json` тАФ it points to Community; verify with `where.exe MSBuild.exe` |
| `CMake Error: Failed to run MSBuild` | Same as above тАФ BuildTools MSBuild cannot compile ARM64 DLL | Install VS 2022 **Community** (not BuildTools); set `VCTargetsPath` to Community path |

## Target architecture selection

Omit `--host-arch`,`--target-arch` to let the script auto-detect (picks the most common case for the host OS).

Override explicitly when you need a specific target:

| `--target-arch` | Purpose |
|----------------|---------|
| `x86_64-linux-clang` | Host testing on x86 Linux (auto-detected default on Linux) |
| `aarch64-ubuntu-gcc9.4` | ARM target device support (cross-compile output) |
| `x86_64-windows-msvc` | Host testing on Windows x86 |
| `windows-aarch64` | **WoS ARM64 target** (auto-detected on WoS) |

> **Note:** `windows-aarch64` produces a DLL for ARM64 Windows deployment.

## Float conversion (FP16/FP32)

```bat
REM Must be called from inside a .bat with VS ARM64 env initialized (or use run_pipeline.bat)
%PYTHON_X64% ${APP_ROOT}\factory\chat_features\model-builder\scripts\qai_convert_fp.py ^
  --onnx ${WORKSPACE}\<model>\<model>.onnx ^
  --precision 16 ^
  --output-root ${WORKSPACE}\<model>\output ^
  --no-simplification
```

The `--no-simplification` flag passes `--no_simplification` to `qnn-onnx-converter`,
which is recommended for WoS ARM64 with QAIRT 2.45.

### Explicit input dimensions for dynamic inputs

If the ONNX model has dynamic input shapes, you must specify explicit dimensions using `--input-dim`:

```bat
REM via wrapper script (comma-separated: name,dim1,dim2,...)
%PYTHON_X64% ${APP_ROOT}\factory\chat_features\model-builder\scripts\qai_convert_fp.py ^
  --onnx ${WORKSPACE}\<model>\<model>.onnx ^
  --precision 16 ^
  --output-root ${WORKSPACE}\<model>\output ^
  --input-dim input,1,3,64,64

REM via direct converter (SPACE-separated between name and dims, comma-separated dims)
%PYTHON_X64% "%QNN_SDK_ROOT%\bin\x86_64-windows-msvc\qnn-onnx-converter" ^
  --input_network model.onnx ^
  --input_dim "input 1,3,512,512" ^
  --output_path output\<model>.cpp ...
```

> тЪая╕П Direct converter format: `--input_dim "name dim1,dim2,dim3"` тАФ **space** between name and dims, **comma** between dims.
> Wrapper script format: `--input-dim name,dim1,dim2,dim3` тАФ all **comma**-separated.

You can specify multiple inputs by repeating the flag:
```bat
%PYTHON_X64% ${APP_ROOT}\factory\chat_features\model-builder\scripts\qai_convert_fp.py ^
  --onnx model.onnx ^
  --input-dim input1,1,3,224,224 ^
  --input-dim input2,1,10
```

## Quantized conversion (INT8/A16W8/A8W8B8)

```bat
REM A16W8 (recommended for vision models)
%PYTHON_X64% ${APP_ROOT}\factory\chat_features\model-builder\scripts\qai_convert_int.py ^

  --weight_bw 8

REM INT8 (A8W8)
%PYTHON_X64% ${APP_ROOT}\factory\chat_features\model-builder\scripts\qai_convert_int.py ^
  --input_network ${WORKSPACE}\<model>\<model>.onnx ^
  --input_list ${WORKSPACE}\<model>\calib\calibration_list.txt ^
  --act_bw 8 ^
  --weight_bw 8

REM A8W8B8 (INT8 with 8-bit bias)
%PYTHON_X64% ${APP_ROOT}\factory\chat_features\model-builder\scripts\qai_convert_int.py ^
  --input_network ${WORKSPACE}\<model>\<model>.onnx ^

Calibration list format (`calibration_list.txt`):
```
input:=${WORKSPACE}\<model>\calib\sample_001.raw

## Context binary generation

Use on target device when host and model-lib arch differ.

```bat
%PYTHON_X64% ${APP_ROOT}\factory\chat_features\model-builder\scripts\qai_dev_gen_contextbin.py ^
  --model C:\absolute\path\to\model.dll ^
  --output_dir output ^
  --binary_file my_model ^
  --auto-config
```

### Options

| Option | Required | Description |
|--------|----------|-------------|
| `--model` | тЬЕ Yes | Absolute path to compiled model library (`.so` on Linux, `.dll` on Windows) |
| `--output` | No | Output path for the context binary. If omitted, output is placed in `output/<name>` |
| `--output_dir` | No | Output directory (used with `--binary_file`) |
| `--binary_file` | No | Output binary name without `.bin` extension |
| `--auto-config` | No | Auto-generate `backend_extensions.json` and `htp_backend_config_v73.json` |
| `--config_file` | No | Path to `backend_extensions.json` (use instead of `--auto-config` for custom config) |
| `--profiling` | No | Enable HTP optrace profiling |

### Backend resolved automatically by script

| Host OS | Arch dir used | Backend |
|---------|--------------|---------|
| Linux | `x86_64-linux-clang` or `aarch64-*` | `libQnnHtp.so` |
| Windows (WoS ARM64) | `aarch64-windows-msvc` | `QnnHtp.dll` |

### Preflight

**Windows** (do not use Python `platform.machine()` or `$env:PROCESSOR_ARCHITECTURE` тАФ both can be affected by emulation):
```powershell
# Reliable: WMI Win32_Processor.Architecture
(Get-WmiObject Win32_Processor).Architecture
# 0 = x86, 5 = ARM, 9 = x64 (AMD64), 12 = ARM64

# Or use dumpbin to check DLL architecture:
dumpbin /headers C:\path\to\model.dll | find "machine"
```

| OS | Host arch | Model lib | Action |
|----|-----------|-----------|--------|
| Linux | `x86_64` | `aarch64` (`.so`) | Blocked on host тАФ run on ARM target device |
| Linux | `aarch64` | `aarch64` (`.so`) | Allowed |
| Windows | `ARM64` | `ARM64` (`.dll`) | Allowed |
| Windows | `AMD64` | `ARM64` (`.dll`) | Blocked on host тАФ run on ARM target device |

If architectures do not match, **do not run** `qnn-context-binary-generator` locally. Stop and instruct the user to run on the target device.

## End-to-End Pipeline (WoS ARM64)

### Default: Flow A тАФ `run_pipeline.py` (ONNX тЖТ DLC тЖТ .bin)

Use `${APP_ROOT}/factory/chat_features/model-builder/scripts/run_pipeline.py` for automated end-to-end conversion.
All logic is in Python тАФ reads `${APP_ROOT}/data/config/qairt_env.json`, invokes `qairt-converter` тЖТ optional `qairt-quantizer` тЖТ `qai_dev_gen_contextbin.py --model <file>.dlc`.
**No VS ARM64 env needed** тАФ the DLC pipeline skips the `.dll` compile step entirely. No `.bat` wrapper needed.

```bat
REM FP16 conversion (default precision)
<python_x64_venv>\Scripts\python.exe ${APP_ROOT}\factory\chat_features\model-builder\scripts\run_pipeline.py ^
  --model ${WORKSPACE}\<model>\<model>.onnx ^
  --output ${WORKSPACE}\<model>\output ^
  --precision fp16

REM bf16 conversion (recommended for Transformer / LLM)
<python_x64_venv>\Scripts\python.exe ${APP_ROOT}\factory\chat_features\model-builder\scripts\run_pipeline.py ^
  --model ${WORKSPACE}\<model>\<model>.onnx ^
  --output ${WORKSPACE}\<model>\output ^
  --precision bf16

REM W8A8 quantization with CLE + per-channel weights + encoding dump (typical B6 fix)
<python_x64_venv>\Scripts\python.exe ${APP_ROOT}\factory\chat_features\model-builder\scripts\run_pipeline.py ^
  --model ${WORKSPACE}\<model>\<model>.onnx ^
  --output ${WORKSPACE}\<model>\output ^
  --precision w8a8 ^
  --calib_list ${WORKSPACE}\<model>\calib\calibration_list.txt ^
  --cle --per_channel --dump_encoding

REM Cross-platform DLC is the DEFAULT (nothing to add). Explicit example:
<python_x64_venv>\Scripts\python.exe ${APP_ROOT}\factory\chat_features\model-builder\scripts\run_pipeline.py ^
  --model ${WORKSPACE}\<model>\<model>.onnx ^
  --output ${WORKSPACE}\<model>\output ^
  --precision w8a16 ^
  --calib_list ${WORKSPACE}\<model>\calib\calibration_list.txt

REM SoC-optimized DLC (only when user explicitly asked тАФ e.g. "tune for X Elite")
<python_x64_venv>\Scripts\python.exe ${APP_ROOT}\factory\chat_features\model-builder\scripts\run_pipeline.py ^
  --model ${WORKSPACE}\<model>\<model>.onnx ^
  --output ${WORKSPACE}\<model>\output ^
  --precision w8a16 ^
  --calib_list ${WORKSPACE}\<model>\calib\calibration_list.txt ^
  --soc_optimized --htp_version v73
```

> ЁЯТб **`run_pipeline.py` is the recommended entry point** тАФ it handles Python paths, HTP runtime file copy, and context binary generation automatically.
> If you need to run individual conversion steps separately (rare тАФ usually only for debugging):
> - **Step 1**  `qairt-converter --input_network model.onnx --output_path model.dlc [--float_bitwidth 16|bf16] [--onnx_skip_simplification]`
> - **Step 2**  `qairt-quantizer --input_dlc model.dlc --output_dlc model_w8a8.dlc --input_list calib.txt --act_bitwidth 8 --weights_bitwidth 8 --use_per_channel_quantization [--algorithms cle]`
> - **Step 3**  `qai_dev_gen_contextbin.py --model model_w8a8.dlc --output_dir output --binary_file model_w8a8 --htp_version v73`
>
> See [`references/context_binary.md`](context_binary.md) for Step 3 details.

### Legacy: Flow C тАФ `run_pipeline_legacy.py` (ONNX тЖТ C++/BIN тЖТ DLL тЖТ .bin)

When the user explicitly asks for the DLL pipeline (or specifically needs the `.dll` artifact), invoke `run_pipeline_legacy.py` at the same script path. Same CLI as the new `run_pipeline.py` (the twelve legacy flags: `--model` / `--output` / `--precision` / `--act_bw` / `--weight_bw` / `--bias_bw` / `--calib_list` / `--input_dim` / `--config` / `--htp_version` / `--skip_contextbin` / `--no_simplification`). Internally it still runs `qnn-onnx-converter` + `qnn-model-lib-generator` (requires VS ARM64 env, initialized internally via `cmd /c vcvarsall.bat arm64 && set`) + `qai_dev_gen_contextbin.py --model <file>.dll`.

For per-step debugging of Flow C, `qai_convert_fp.py` / `qai_convert_int.py` are still the wrapper scripts (see the Float / Quantized conversion sections above).

### `run_pipeline.py` тАФ Full Argument Reference

**Legacy-compatible flags** (identical semantics to the old `run_pipeline.py`):

| Argument | Required | Default | Description |
|----------|----------|---------|-------------|
| `--model` | Yes | тАФ | Path to input ONNX model |
| `--output` | No | `qairt_output` | Output directory |
| `--precision` | No | `fp16` (if neither `--precision` nor `--act_bw`/`--weight_bw` is given) | `fp32` \| `fp16` \| `bf16` \| `w4a8` \| `w4a16` \| `w8a8` \| `w8a8b8` \| `w8a16` \| `w16a16`. Mutually exclusive with `--act_bw`/`--weight_bw`. |
| `--act_bw` | No* | тАФ | Custom activation bitwidth (4/8/16). Requires `--weight_bw`. |
| `--weight_bw` | No* | тАФ | Custom weight bitwidth (4/8/16). Requires `--act_bw`. |
| `--bias_bw` | No | тАФ | Custom bias bitwidth (8/32). Unspecified тЗТ bias stays FP32 (legacy semantics preserved via internal `--float_bias_bitwidth 32`). |
| `--calib_list` | Yes (quantized) | тАФ | Calibration list file path. **Raw paths, one per line тАФ no `input:=` prefix.** |
| `--input_dim` | No | тАФ | Input dimensions, e.g. `"input 1,3,512,512"`. Passed to `qairt-converter --source_model_input_shape`. |
| `--config` | No | тАФ | Path to HTP `backend_extensions.json` (optional). |
| `--htp_version` | No | `v73` | HTP version for context binary: `v73` (X Elite / SC8380XP) or `v81` (X2 Elite / SM8750). |
| `--skip_contextbin` | No | False | Skip context binary generation (output DLC only) |
| `--no_simplification` | No | False | Skip ONNX simplification. Transparently maps to `qairt-converter --onnx_skip_simplification`. |

**New DLC-path flags** (all optional; do not exist in the legacy pipeline):

| Argument | Default | Description |
|----------|---------|-------------|

*`--act_bw` and `--weight_bw` must be used together.


**Output naming convention:** Context binary is named `{model_stem}_{precision}.bin`
- e.g., `inception_v3_fp16.bin`, `inception_v3_int8.bin`
